# Phase 03.1: Empirical Model Realignment - Research

**Researched:** 2026-03-11
**Domain:** Active v1 model recalibration inside the existing calculator contract
**Confidence:** HIGH

<user_constraints>
## User Constraints

### Required outcomes
- Keep one active v1 model in product UX. No user-facing model selector in this phase.
- Keep the live calculator flow intact. Phase 03.1 is a model-truth correction, not a UI redesign.
- Keep the source of truth as a simple ambient-use time+temperature formula, not a runtime table.
- Use the social table and Calcolapizza only as calibration evidence or comparison fixtures, not as the official runtime model.
- Improve practical trust, especially at short fermentation times, without introducing cliffs or strange behavior.
- Preserve the current domain/result contract unless a change is strictly necessary for calibration integrity.

### Phase boundary
- This phase should focus on `lib/calc` model truth, its internal seams, and numeric validation.
- It should not absorb Phase 4 work such as premium presentation, formula storytelling, or disclaimer layout.
- It should not absorb Phase 5 work such as active advanced modifiers.
- It may add internal architecture that makes future multi-model work easier, but that seam must stay hidden in v1 UX.
</user_constraints>

<research_summary>
## Summary

Phase 03.1 should be planned as a **numerical realignment phase with a contract-preserving implementation seam**, not as a calculator rewrite.

The current model is:

`gramsPerKg = 14.5 * timeHours^-0.89 * 2^-((temperatureC - 20) / 10)`

This is mathematically clean, but it has two hard-coded behaviors that matter for planning:
- doubling fermentation time always divides yeast by about `1.853`, no matter whether the user is moving from `2 -> 4 h` or `12 -> 24 h`
- increasing temperature by `10C` always halves yeast, no matter the time

That separable constant-slope behavior is the likely reason short-time outputs can feel wrong in practice. It is coherent, but it is too rigid if the trust gap is concentrated in the lower-time region.

**Primary recommendation:** plan the phase around a two-step decision gate:

1. Build a comparison-fixture corpus first and score the existing model against it.
2. Try a coefficient retune inside the current formula family first.
3. If that cannot fix short-time trust without harming the rest of the range, move to a **piecewise but still simple formula family** that remains smooth, monotonic, and explainable.

The safest planning default is:
- preserve `calculateFreshYeast` and the result shape as-is
- keep warnings, rounding, normalization, and UI contracts unchanged unless calibration evidence forces a range update
- isolate model truth behind a small internal seam in `lib/calc/model.ts`
- convert model tests from "assert literal old formula" to "assert calibration fixtures + invariants"

This phase succeeds if the calculator keeps behaving the same structurally while the numeric source of truth becomes more credible.
</research_summary>

## Current Baseline

### Current implementation facts
- `lib/calc/model.ts` contains one closed-form formula plus linear flour scaling.
- `lib/calc/calculate-fresh-yeast.ts` already cleanly separates normalization, validation, base-model execution, warnings, rounding, and result shaping.
- `lib/calc/validation.ts` owns the non-blocking empirical warning based on fixed temperature/time ranges.
- `components/calculator/interactive-calculator.tsx` and `components/calculator/calculator-result-panel.tsx` consume only the final calculation result, not the formula internals.
- Existing integration tests call `calculateFreshYeast` to derive expected UI values, so the UI surface is already loosely coupled to the formula details.

### Current regression-sensitive tests
- `tests/unit/calc/model.test.ts` is tightly coupled to the literal current formula and constants.
- `tests/unit/calc/calculate-fresh-yeast.test.ts` is mostly contract-oriented, but its happy-path expectation still derives from the same base helper, so it is not independent evidence.
- `tests/unit/calc/validation.test.ts` and `tests/integration/home-page.test.tsx` are sensitive only if empirical ranges or warning copy change.

### Current model behavior over representative valid inputs
At `20C`, the current curve produces approximately:

| Time | Current output |
|------|----------------|
| 2 h | 7.824 g/kg |
| 4 h | 4.222 g/kg |
| 6 h | 2.943 g/kg |
| 8 h | 2.278 g/kg |
| 12 h | 1.588 g/kg |
| 24 h | 0.857 g/kg |

At fixed time, every `+10C` halves the output.

Planning implication:
- if the practical trust problem is local to short times, changing only the coefficient may not be enough
- the main question is whether the current formula family is merely mis-tuned or structurally too rigid

<architecture_patterns>
## Architecture Patterns

### Pattern 1: Keep the public calculator contract stable
The strongest default is to keep all of this unchanged:
- `calculateFreshYeast(input: CalculatorInput): CalculationResult`
- `CalculationResult` discriminated union shape
- `roundPracticalYeast`
- normalization flow and flour scaling
- warning semantics: valid out-of-range inputs still calculate
- UI consumption pattern: result panel receives final numbers plus one warning string

That keeps Phase 03.1 isolated to model truth and avoids accidental scope creep into the already-complete calculator flow.

### Pattern 2: Insert a small internal model-definition seam
The current formula is inlined directly in `calculateBaseYeastPerKg`. Planning should likely introduce an internal seam such as:

```ts
type EmpiricalModelDefinition = {
  calculate(params: { temperatureC: number; timeHours: number }): number;
  reliabilityRange: {
    temperatureC: { min: number; max: number };
    timeHours: { min: number; max: number };
  };
};
```

Recommended rule:
- keep the seam internal to `lib/calc`
- keep `calculateBaseYeastPerKg` as the stable exported entrypoint
- let `calculateBaseYeastPerKg` delegate to the active internal model definition

This preserves the current external shape while making it possible to swap from "single power law" to "piecewise simple family" without rewriting the rest of the pipeline.

### Pattern 3: Separate fit parameters from public constants
Right now the formula constants are exported through the barrel:
- `FORMULA_BASE_COEFFICIENT`
- `FORMULA_TIME_EXPONENT`
- `FORMULA_REFERENCE_TEMPERATURE_C`
- `FORMULA_TEMPERATURE_STEP_C`

That is harmless for the current implementation, but brittle for realignment. If Phase 03.1 adopts a piecewise family, these names become misleading.

Planning recommendation:
- keep empirical warning ranges public if the app needs them
- move fit coefficients into an internal config/module unless they are intentionally part of the public product contract
- avoid designing tests that depend on raw coefficient exports

### Pattern 4: Use calibration fixtures as the new source of test truth
This phase should replace "formula equals old formula" tests with:
- comparison fixtures
- invariant tests
- continuity tests if a piecewise family is used

The runtime source of truth remains a formula. The **test source of truth** becomes curated comparison evidence.
</architecture_patterns>

<implementation_options>
## Implementation Options

### Option A: Retune coefficients inside the same formula family

Family:

`A * timeHours^B * 2^-((temperatureC - T_ref) / step)`

What changes:
- coefficient `A`
- time exponent `B`
- temperature step and possibly reference temperature
- possibly the empirical reference range

Pros:
- lowest implementation cost
- keeps one compact closed form
- guarantees smoothness and monotonicity if coefficients remain sensible
- minimal code churn in `lib/calc/model.ts`
- easiest formula story for Phase 4

Cons:
- cannot change the fact that time elasticity is constant across the full range
- cannot make short-time behavior more generous without also shifting medium/long times in a similar structural way
- may fail if the trust gap is specifically "too harsh below about 6-8 hours"

When this is enough:
- the fixture set shows a mostly uniform bias across the valid range
- short-time scenarios are off by degree, not by shape

Planning note:
- this should be the first attempted path because it is cheap to test
- the plan should include an explicit accept/reject gate based on fixture error bands

### Option B: Piecewise time family that stays simple

Example shape:
- one branch for short times
- one branch for standard times
- same temperature factor across both
- continuity enforced at the boundary

Possible families:
- two power-law branches with matched value at the boundary
- short-time correction multiplier that decays toward `1` after a chosen transition point
- smooth blend between two simple branches across a narrow transition window

Pros:
- directly targets the actual reported trust problem
- allows short-time generosity without distorting the whole long-time curve
- still explainable as a simple formula family if done carefully
- keeps runtime cheap and deterministic

Cons:
- planning must explicitly guard against cliffs at the branch boundary
- tests need continuity and local-slope coverage, not only spot checks
- formula transparency in Phase 4 becomes slightly more complex

When this is warranted:
- coefficient-only retuning cannot hit short-time fixtures without noticeably harming the rest of the range
- the calibration corpus shows different behavior regimes instead of one uniform bias

Planning note:
- this is the most likely best option if the trust issue is concentrated in short fermentations
- the branch should be based on `timeHours`, not on hidden UI state or arbitrary lookup tables

### Option C: Runtime lookup table or interpolation family

Examples:
- promote the social table to active runtime truth
- interpolate across calibration points at runtime
- store temperature-time cells and derive output from nearest or bilinear lookup

Pros:
- easy to match fixtures exactly

Cons:
- violates the phase direction that v1 should remain a simple formula, not a table as official truth
- weakens future transparency because the product would claim a formula while effectively using a table
- increases hidden complexity without user-facing value in this phase

Recommendation:
- reject as the active runtime model for Phase 03.1
- allow lookup tables only as calibration evidence in tests or fitting notebooks/scripts

### Option D: Broader multi-model registry now

Pros:
- could make future multi-model work easier

Cons:
- unnecessary product-scope broadening
- risks leaking future architecture into the current phase
- does not solve the calibration question by itself

Recommendation:
- do not plan a visible registry or selector
- at most, add a tiny internal seam that supports one active model only
</implementation_options>

<recommended_approach>
## Recommended Approach

### Prescriptive recommendation
Plan Phase 03.1 around **fixture-led recalibration with a contract-preserving internal seam**.

Recommended sequence:

1. Create a calibration fixture set that represents the "practical truth" the team is willing to defend for standard ambient use.
2. Score the current model against that fixture set and identify where the error clusters.
3. Attempt coefficient retuning in the existing separable family first.
4. If coefficient retuning cannot bring short-time scenarios into an acceptable trust band, implement a piecewise time family with enforced continuity and preserved monotonicity.
5. Keep the rest of the calculator pipeline unchanged.

### Why this is the best planning default
- It respects the explicit decision to keep a simple time+temperature model.
- It avoids silently turning the product into a table-driven calculator.
- It creates a concrete decision gate instead of guessing whether a more complex family is needed.
- It keeps the already-working calculator UI and orchestration stable.

### What not to hand-roll
- Do not invent a large fitting framework inside the app runtime.
- Do not move calibration evidence into UI code.
- Do not make warnings or rounding depend on formula branch details.
- Do not preserve old formula-constant exports if they stop describing reality.
</recommended_approach>

<planning_inputs>
## What Planning Needs To Decide Explicitly

### 1. Calibration corpus
Planning must define what evidence the model is trying to match.

Recommended artifact:
- a small fixture corpus owned by the repo
- each fixture includes `temperatureC`, `timeHours`, expected `gramsPerKg`, source label, and rationale
- label whether the case is core-range, boundary, or comparison-only

Recommended shape:

```ts
type ModelCalibrationFixture = {
  id: string;
  temperatureC: number;
  timeHours: number;
  expectedGramsPerKg: number;
  source: 'social-table' | 'calcolapizza' | 'manual-target';
  weight: 'high' | 'medium' | 'low';
  notes?: string;
};
```

Planning implication:
- short-time fixtures should be intentionally overrepresented if that is the known trust problem
- fixtures should stay inside the standard ambient-use window; out-of-range cases belong to warning tests, not fit targets

### 2. Acceptance criteria for model fitness
Planning should not say only "make it more credible."

It should define:
- max tolerated error for high-trust fixtures
- separate tolerance for comparison-only fixtures
- whether fitness is judged by absolute error, relative error, or both
- whether short-time fixtures are weighted more heavily

Good planning default:
- use relative error for most fixtures
- use slightly stricter weighting for short-time core-range cases

### 3. Decision gate for staying in one closed form vs moving piecewise
Planning needs an explicit trigger such as:
- if coefficient-only retuning cannot satisfy the agreed fixture tolerances without materially worsening medium/long-time cases, escalate to a piecewise time family

Without this gate, the phase can drift into endless subjective tweaking.

### 4. Boundary and continuity rules if piecewise is used
If a piecewise family is chosen, planning must specify:
- boundary variable: `timeHours`
- boundary location or transition window
- continuity requirement at the boundary
- monotonicity requirement on both sides
- whether derivative smoothness is required or value continuity is sufficient

Default recommendation:
- require value continuity
- prefer smooth blending if it remains easy to reason about
- do not require a mathematically perfect derivative match if that complicates the model beyond the phase goal

### 5. Reliability range ownership
If calibration reveals that the current `16-36C` and `2-24 h` window is too broad or too narrow, planning must decide whether the empirical warning range changes in this phase.

Rule:
- only change the range if the calibration evidence clearly supports it
- if the range changes, plan updates to constants, warning copy, validation tests, and integration assertions together

### 6. Public-vs-internal API posture
Planning should decide whether old formula coefficient exports remain public.

Default recommendation:
- keep warning-range constants public if needed
- make fit parameters internal
- keep `calculateBaseYeastPerKg` public as the stable model entrypoint
</planning_inputs>

<common_pitfalls>
## Common Pitfalls

### Pitfall 1: Treating calibration evidence as runtime truth
Using the social table as fixtures is fine. Turning it into the hidden runtime engine violates the phase intent.

### Pitfall 2: Retuning coefficients without first defining acceptance criteria
Without explicit fixture tolerances, model tuning becomes subjective and hard to finish.

### Pitfall 3: Fixing short times by introducing discontinuities
A hard branch at `6 h` or `8 h` with no continuity check can create visible cliffs that damage trust more than the original bias.

### Pitfall 4: Keeping formula-specific exports public after the model family changes
If the code still exports old coefficient names after adopting a piecewise family, tests and future phases will build on a false abstraction.

### Pitfall 5: Expanding scope into explanation UI
Phase 03.1 should make Phase 4 safer. It should not absorb the formula-visibility or premium-copy work itself.

### Pitfall 6: Confusing warning reliability with fit quality
The warning range is about where the app claims confidence. The fit fixture set is about how accurate the active model is within that confidence window. These are related but not identical.

### Pitfall 7: Losing independent regression coverage
If every expected value is recomputed from the implementation under test, the phase can ship a coherent but still untrusted model.
</common_pitfalls>

<validation_architecture>
## Validation Architecture

### Validation goals
The planner should create `03.1-VALIDATION.md` around three layers:
- model-fit validation
- mathematical-behavior validation
- calculator-contract regression validation

### Layer 1: Model-fit validation
Owner: unit tests and any fitting helper script

Purpose:
- prove the active model stays within agreed tolerances against the curated fixture corpus
- make short-time trust measurable instead of subjective

Recommended artifacts:
- fixture file under `tests/unit/calc/fixtures/`
- optional local helper script for scoring candidate coefficient sets

Recommended checks:
- table-driven assertions against expected `gramsPerKg`
- weighted reporting for short-time fixtures
- explicit failure output that shows worst offending scenarios

### Layer 2: Mathematical-behavior validation
Owner: unit tests in `tests/unit/calc/model.test.ts`

Purpose:
- prevent "fixes" that create numerically ugly behavior

Recommended checks:
- monotonicity: higher temperature should not increase required yeast at fixed time
- monotonicity: longer time should not increase required yeast at fixed temperature
- continuity at branch boundaries if piecewise is used
- no local cliffs around the short-time transition
- stable scaling invariants: flour scaling remains linear and separate from base-model math

Recommended extra check for piecewise families:
- compare values just below, at, and just above the branch boundary or blend window

### Layer 3: Calculator-contract regression validation
Owner: existing orchestrator, integration, and app tests

Purpose:
- prove the model realignment did not break the already-complete calculator flow

Required checks:
- valid input still returns `status: 'ok'`
- invalid input still short-circuits to `status: 'invalid'` with `warnings: []`
- out-of-range but valid input still returns one warning and visible results
- UI still updates live and suppresses stale output on invalid edits
- result formatting and rounding behavior stay unchanged unless intentionally revised

### Recommended command posture
Even if implementation is local to `lib/calc`, phase closeout should still run:

```bash
npm run test:unit
npm run test:integration
npm run lint
npm run typecheck
npm run build
```

If a fitting helper script is added for local calibration, it should not become a required production dependency.

### Recommended test split
- `model.test.ts`: fixture fit + invariants + continuity
- `calculate-fresh-yeast.test.ts`: orchestration contract still intact
- `validation.test.ts`: warning-range semantics if the reliability window changes
- `home-page.test.tsx`: UI contract remains intact with new model truth
</validation_architecture>

## Code Examples

### Minimal contract-preserving model seam

```ts
type BaseModelParams = {
  temperatureC: number;
  timeHours: number;
};

type EmpiricalModelDefinition = {
  calculate(params: BaseModelParams): number;
};

const activeEmpiricalModel: EmpiricalModelDefinition = {
  calculate({ temperatureC, timeHours }) {
    return calculateRealignedYeastPerKg({ temperatureC, timeHours });
  },
};

export function calculateBaseYeastPerKg(params: BaseModelParams): number {
  return activeEmpiricalModel.calculate(params);
}
```

### Example piecewise direction that stays simple

```ts
function calculateRealignedYeastPerKg({
  temperatureC,
  timeHours,
}: BaseModelParams): number {
  const temperatureFactor = Math.pow(2, -((temperatureC - 20) / 10));

  if (timeHours <= SHORT_TIME_BOUNDARY_HOURS) {
    return SHORT_A * Math.pow(timeHours, SHORT_B) * temperatureFactor;
  }

  return LONG_A * Math.pow(timeHours, LONG_B) * temperatureFactor;
}
```

Planning rule:
- only choose this direction if continuity is explicitly enforced and fixture evidence justifies it
- otherwise prefer a single retuned closed form
</code_examples>

<planner_takeaway>
## Planner Takeaway

The key planning question is not "how do we tweak the formula a bit?" It is:

**Can the current separable power-law family be recalibrated enough to restore trust at short times, or is the short-time issue structural and therefore better solved with a simple piecewise time family?**

The phase should be planned to answer that question decisively through fixtures and acceptance gates, while keeping the calculator contract and UX stable.

If planning stays disciplined, Phase 03.1 can be small in surface area but high in trust impact.
</planner_takeaway>
