# Phase 03.1: Empirical Model Realignment - Context

**Gathered:** 2026-03-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Realign the active v1 calculation model so the product keeps one trustworthy estimator, but with a source of truth that better matches practical use. This phase should revisit the base math and its calibration, while keeping the current live-calculator flow intact and leaving premium presentation, formula storytelling, and any user-facing multi-model choice to later phases.

</domain>

<decisions>
## Implementation Decisions

### Active v1 model strategy
- The v1 remains mono-model in product UX.
- The active v1 model should remain a simple time-plus-temperature formula rather than a table becoming the official truth of the product.
- The social table can be used as calibration evidence or as a comparison check, but not as the authoritative source of truth by itself.
- The v1 model should optimize for standard ambient use, without fridge stages or advanced variables.

### Calibration priorities
- Practical adherence matters more than theoretical elegance.
- The phase should protect trust by avoiding strange slopes, abrupt behavior changes, or outputs that feel disconnected from kitchen intuition.
- On short fermentations, the recalibrated model may lean slightly more generous rather than systematically underestimating yeast.
- The product should not try to converge toward external calculators just because they exist; credibility comes from declared assumptions, not imitation.

### Reliability window and edge behavior
- The official reliability window should be a practical "cucina standard" range: neither so narrow that warnings fire too early nor so wide that the model overclaims.
- Outside the model's best-fit zone, behavior should stay regular and readable rather than introducing sudden cliffs or special-case jumps.
- Existing warning semantics remain in force: valid but out-of-range input still returns a result with a sober non-blocking warning.
- The phase should improve the model truth, not redesign the result-state UX established in Phase 3.

### Claude's Discretion
- Exact formula family, fitting approach, and target calibration data as long as the result remains a simple ambient-use model.
- Exact reference-range boundaries and how they are encoded in warnings.
- Exact test strategy needed to prove the new model is numerically credible and does not break the current calculator contract.

</decisions>

<specifics>
## Specific Ideas

- The current formula implementation is internally coherent, but it has already lost trust because some short-time outputs diverge too much from practical expectations.
- The social table and Calcolapizza are useful comparative artifacts, but neither should be silently promoted to official truth for the v1.
- This phase should make the next presentation phase safer by ensuring the math being explained is the math the product is willing to stand behind.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `lib/calc/model.ts`: currently contains the single closed-form base formula and flour scaling seam that this phase will likely recalibrate or replace internally.
- `lib/calc/calculate-fresh-yeast.ts`: already separates normalization, validation, base-model execution, rounding, warnings, and result shaping.
- `tests/unit/calc/`: provides a strong place to lock the new model behavior with numeric assertions and comparison fixtures.
- `components/calculator/interactive-calculator.tsx` and `components/calculator/calculator-result-panel.tsx`: the mounted calculator flow already exists, so the model realignment should avoid unnecessary UI churn.

### Established Patterns
- Phase 2 established a pure, reusable domain contract and a sober warning model.
- Phase 3 established that stale results are not acceptable and that valid out-of-range input still renders a result.
- The product direction is now explicit: one active v1 model, with future multi-model readiness hidden behind the architecture rather than exposed in the UX.

### Integration Points
- Phase 03.1 should preserve the public calculation contract consumed by the current page and calculator components.
- Any future model-registry seam should stay internal in this phase; the user should not see model selection yet.
- The new model truth must remain compatible with the current validation, warning, rounding, and preset flow unless a clear planning decision says otherwise.

</code_context>

<deferred>
## Deferred Ideas

- User-facing selection between multiple declared models belongs to a later milestone or follow-up phase.
- Formula visibility, disclaimer layout, and trust-centered presentation still belong to Phase 4.
- Advanced variables such as hydration, fridge staging, FDT, or pasta di riporto remain out of scope for this realignment phase.

</deferred>
